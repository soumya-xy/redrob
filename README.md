# Redrob Hackathon — Intelligent Candidate Discovery & Ranking

This repository contains a candidate ranking system for the
Senior AI Engineer role described in `job_description.txt`.

## What it does

Given a pool of ~100,000 candidate profiles, produces a top-100 ranked CSV
where rank 1 is the best fit for the JD, with a 1-2 sentence reasoning per row.

## Architecture

Two-phase pipeline. The hard 5-minute wall is on the **online ranking step**
that produces the CSV. Offline precomputation can take any length of time.

```
OFFLINE (~25 min, no time limit)
  precompute.py
    1. Load 100k candidates from candidates.jsonl
    2. Extract 54 engineered features per candidate
    3. Compute credibility score (honeypot safety net)
    4. Encode all 100k profiles with sentence-transformer (MiniLM-L6, 384-dim)
    5. Build BM25 index over profile+career text
    6. Encode JD text
    7. Build centroid from 10 hand-picked reference candidates
    8. Persist all artifacts to artifacts/

ONLINE (~90 sec, must be < 5 min on CPU)
  produce_submission.py
    1. Load artifacts
    2. Hybrid retrieval: BM25 + dense-JD + dense-centroid
    3. Force-include the 10 centroid picks in the top-500 funnel
    4. Score top 500 with weighted feature sum
    5. Apply availability multiplier (multiplicative, not additive)
    6. Apply credibility multiplier (honeypot safety net)
    7. Take top 100
    8. Generate per-row reasoning from feature-conditioned templates
    9. Mechanical no-hallucination check on each reasoning
   10. Write team_xxx.csv
```

## Files

- `precompute.py` — offline precomputation
- `produce_submission.py` — online ranking step
- `reasoning.py` — reasoning template library + hallucination checker
- `select_reference_candidates.py` — utility to scan candidates and produce
  a shortlist for hand-picking the centroid
- `centroid_picks.json` — the 10 hand-picked reference candidates with
  JD-criterion justifications (used as Stage 4 audit evidence)
- `shortlist.json` — 50-candidate shortlist (raw output of the picker)
- `team_xxx.csv` — final submission
- `artifacts/` — precomputed indexes and features (regenerated by precompute.py)
- `submission_metadata.yaml` — metadata for the submission portal

## Reproducing the submission

The two-command reproduction:

```bash
# 1. Offline precomputation (~25 min on CPU)
python precompute.py

# 2. Online ranking step (~90 sec on CPU, must be < 5 min)
python produce_submission.py --out team_xxx.csv
```

## Design decisions and tradeoffs

### Ranker: weighted feature sum (not LightGBM)
- No labeled training data, so a learned ranker would overfit to a small
  hand-labeled set or to its own pseudo-labels
- A weighted sum is interpretable: every weight has a JD-anchored justification
- Easier to defend at Stage 5 (defend-your-work interview)

### Hybrid retrieval: BM25 + dense-JD + dense-centroid
- BM25 catches exact keyword matches ("Pinecone", "RAG")
- Dense-JD catches semantic matches ("retrieval system" → "candidate sourcing")
- Dense-centroid is the Tier 5 fix: candidates whose career trajectory matches
  the JD's "shipped a recommendation system" line, but who don't use the
  literal words in their profile, are recovered by similarity to the centroid

### Centroid: 10 hand-picked, deliberately diverse
- Picked 10 from a 50-candidate shortlist, spanning JD criteria (embedding
  search, ranking/eval, recsys, LLM fine-tune, ML infra), geography
  (Pune/Noida preferred, plus tier-1 Indian cities), seniority (1 Staff,
  8 Senior/Lead, 1 from Netflix for global-tier anchor), and company type
  (mix of big-tech and Indian product companies)
- Documented in `centroid_picks.json` with per-pick JD-criterion justification
  (used as Stage 4 evidence of deliberate, defensible choice)

### Activity signals: multiplicative availability factor
- Per the JD: "perfect-on-paper candidate who hasn't logged in for 6 months
  and has a 5% response rate is, for hiring purposes, not actually available.
  Down-weight them appropriately."
- Applied as `score *= days_active_curve * response_rate_curve * open_to_work`
- days_active: linear decay from 30→90 days, heavy decay 90→180, severe >180
- response_rate: sqrt curve (0%→0.5x, 50%→0.85x, 100%→1.0x)
- open_to_work: 1.0 if True, 0.7 if False

### Credibility engine: honeypot safety net
- 6-rule check per candidate: years-vs-career-mismatch, expert-no-duration
  skills, too-many-expert-skills, date inconsistency, overlap, current-company
  mismatch
- Output 0-1 score, applied as multiplicative factor (clipped at 0.3 min)
- README says "you don't need to special-case honeypots" but a hard floor
  at low credibility is cheap insurance against the 10% top-100 threshold

### Reasoning: feature-conditioned templates with mechanical hallucination check
- 3 template banks: STRONG (rank 1-30), MID (31-70), LOW (71-100)
- Templates use feature conditions (years, top skills, location, notice period)
- Per-reasoning hallucination check: extract capitalized multi-word phrases,
  verify they're either in candidate's skills/career text or in a whitelist
  of known terms (company names, cities, common role nouns)
- ~71% of reasonings pass on first try; 29% fall back to a generic message
  (still facts-correct, just less specific)

## Validation

```bash
python validate_submission.py team_xxx.csv
# Output: "Submission is valid."
```

## Compute compliance

The online ranking step runs in ~90 seconds on CPU with <2 GB RAM,
well within the 5-min / 16 GB / CPU-only / no-network constraints.

candidate sample (≤100) for the sandbox demo.
